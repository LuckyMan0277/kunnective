## 1. Default Workflow: Haiku 작성 / Opus 검토 / Sonnet 조사

이 레포에서 메인 에이전트는 직접 작업하지 않고 오케스트레이터 역할만 한다. 작업 유형별로 다음 모델에 위임한다:

- **작성(code + docs)**: Haiku → Opus 검토 4단계 루프 (이 섹션)
- **조사·리서치**: Sonnet 단독 위임 (§2 참조)

작성 루프(이 섹션)의 4단계는 다음과 같다.

### Step 1 — Writer (Haiku)

`Agent` 도구로 작성을 위임:

- `subagent_type`: `"general-purpose"`
- `model`: `"haiku"`
- `prompt`: 아래 **Haiku 위임 템플릿** 사용

Haiku 서브에이전트는 실제 파일 편집(Write/Edit)을 수행한다.

### Step 2 — Reviewer (Opus)

Haiku가 끝나면 즉시 검토를 위임:

- `subagent_type`: `"general-purpose"`
- `model`: `"opus"`
- `prompt`: 아래 **Opus 검토 템플릿** 사용

Opus는 변경된 파일을 직접 읽고(`Read`/`Bash` git diff 등) 검토한다.

### Step 3 — Loop until PASS

Opus 응답의 첫 줄이 정확히 `PASS`가 아니면:

1. Opus 피드백을 Haiku에 다시 위임 → 수정
2. 다시 Opus에 검토 위임
3. `PASS`가 나올 때까지 반복 (상한 없음)

**안전장치**: 동일한 이슈가 3회 연속 해결되지 않으면 루프를 멈추고 사용자에게 상황을 보고한 뒤 판단을 구한다.

### Step 4 — Final Report

`PASS` 후 메인 에이전트가 사용자에게 1~2문장으로 변경 요약을 보고한다.

## 2. Scope

### 적용 대상 (반드시 위 루프 사용)

- 모든 코드 파일 작성/수정 (`.ts`, `.tsx`, `.js`, `.py`, `.sql`, …)
- 모든 문서 작성/수정 (`*.md`, `README`, 스펙 문서)
- 모든 설정/스키마 파일 (`package.json`, `tsconfig.json`, 마이그레이션, 환경 설정 등)

한 줄 수정·오타 수정도 예외 없이 루프를 따른다.

### 메인 에이전트가 직접 처리

- 단순 파일 읽기/grep/ls 등 한두 번의 셸 호출로 끝나는 read-only 조회
- 사용자 질문에 대한 답변 (구현이 아닌 설명)
- git 상태 확인 등 read-only 셸 명령
- 사용자가 명시적으로 "직접 해줘", "검토 생략" 등을 요청한 경우

### Sonnet에 위임 (Researcher)

다음 류는 `Agent` 도구를 `subagent_type: "general-purpose"`, `model: "sonnet"`으로 호출해 위임한다:

- **웹 검색·웹 페이지 fetch**: WebSearch / WebFetch 사용이 필요한 모든 조사
- **외부 자료 요약·비교**: 라이브러리 문서, 게임 레퍼런스, 디자인 자료 등 수집
- **코드베이스 광범위 탐색**: 3회 이상 grep/find가 필요할 만한 구조 분석 (Explore 에이전트와 동등)
- **사실 확인이 필요한 답변**: 사용자 질문이 외부 사실/현재 시점 정보에 기반할 때

Sonnet 위임 결과는 메인 에이전트가 받아 사용자에게 정리해 보고하거나, 후속 작성 작업의 입력으로 사용한다. **조사 결과 자체는 Opus 검토 대상이 아님** — 이를 바탕으로 한 코드/문서 변경 단계에서 비로소 Haiku→Opus 루프를 탄다.

## 3. Pass Criteria

Opus는 다음을 **모두** 만족해야 `PASS`를 반환한다:

- 사용자 요구사항을 빠짐없이 충족
- 기능적 정확성 (엣지 케이스, 에러 경로 포함)
- 보안 이슈 없음 (OWASP Top 10 관점: injection, XSS, 인증/인가, 비밀 노출 등)
- 기존 코드 패턴/유틸 재사용, 불필요한 중복 없음
- 군더더기 없음: 불필요한 주석, 과한 추상화, 사용되지 않는 코드, 하위호환 hack 부재
- 문서의 경우: 사실 정확성, 일관된 톤, 명확성

하나라도 미흡하면 `FAIL`을 반환하고 구체적인 수정 지시를 첨부한다.

## 4. Delegation Prompt Templates

### Haiku 위임 템플릿

```
작업: <한 문장 목표>

관련 파일:
- <절대 경로 1>
- <절대 경로 2>

요구사항:
- <불릿 1>
- <불릿 2>

제약:
- <스타일/패턴/금지사항>
- 기존 유틸/패턴이 있으면 재사용할 것
- 불필요한 주석·과한 추상화 금지

산출물:
- 변경/생성할 파일 경로와 기대 결과
- 변경 후 git diff를 한 번 출력해 보고할 것
```

### Opus 검토 템플릿

```
원래 사용자 요구사항:
<사용자 원문 그대로>

Haiku가 수행한 변경:
- <파일 경로>: <간단 요약>
- (필요시 핵심 코드 또는 diff 발췌)

검토 기준:
- 요구사항 충족
- 기능적 정확성 (엣지 케이스 포함)
- 보안 (OWASP Top 10)
- 기존 패턴/유틸 재사용
- 군더더기 제거 (주석, 추상화, 미사용 코드, 하위호환 hack)
- 문서면 사실 정확성·톤·명확성

검토 방법:
- 변경 파일을 직접 Read로 확인
- 필요시 git diff, 관련 파일 탐색

응답 형식 (반드시 준수):
- 첫 줄: "PASS" 또는 "FAIL"
- FAIL인 경우 본문에 구체 지시:
  - 파일 경로 + 라인
  - 무엇이 문제인지
  - 어떻게 고칠지
  - 왜 그렇게 고쳐야 하는지
- PASS인 경우 본문에 한 줄 요약 (선택)
```

### Sonnet 조사 위임 템플릿

```
조사 목표:
<한 문장 — 알아내려는 것>

배경/맥락:
- <왜 필요한지 짧게>

수집 항목:
- <불릿 1: 구체적인 정보 단위>
- <불릿 2>

제약:
- 신뢰 가능한 출처 우선 (공식 문서/스토어 페이지/위키)
- 추측 금지, 출처 URL 함께 보고
- 결과는 메인이 사용자에게 정리해 전달하므로 200~400자 내 요약 + 구조화된 항목

산출물 형식:
- 짧은 요약 1~2문장
- 핵심 항목 불릿 정리
- 참고 URL 목록
```

## 5. Notes

- 메인 에이전트의 모델은 무엇이든 상관없다. 이 워크플로우에서 메인은 오케스트레이터일 뿐.
- 서브에이전트 위임 프롬프트는 항상 **자기완결적(self-contained)** 으로 작성한다. 서브에이전트는 이 대화의 이전 맥락을 보지 못한다.
- 병렬화 가능한 독립 작업은 여러 Haiku를 동시에 호출해도 된다. 단, 각각의 산출물에 대해 Opus 검토는 반드시 수행한다.
- Haiku/Opus/Sonnet 호출은 비용이 발생한다. 단순 read-only 작업은 메인이 직접 처리하고, 외부 조사가 필요할 때만 Sonnet, 작성·수정이 필요할 때만 Haiku→Opus 루프로 가서 불필요한 위임을 피한다.

## 6. General Coding Standards

서브에이전트들이 따라야 할 공통 원칙:

- 기존 파일 편집을 새 파일 생성보다 우선
- 요청된 범위를 넘는 리팩터·추상화·기능 추가 금지
- 내부 코드/프레임워크 보장은 신뢰하고, 시스템 경계(사용자 입력, 외부 API)에서만 검증
- 주석은 *왜*가 비자명할 때만. *무엇*을 설명하지 않는다
- 한국어/영어 혼용 시 사용자 메시지 톤을 따른다
